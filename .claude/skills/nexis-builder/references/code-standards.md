# Code Standards (stack-agnostic)

These apply to all code generated by any phase. They are written to hold regardless of the stack the blueprint chose. Where a rule names a language feature, apply the equivalent in the actual stack.

## The one rule that governs the rest

Match the blueprint's stack and the ecosystem's idioms. Do not impose patterns from a different stack. Idiomatic Django is not idiomatic Next.js. Read the framework's own conventions first, then write code that looks like it belongs.

## Language & tooling

- Use the language's strict/typed mode where it exists (TypeScript strict, Python type hints + a checker, etc.). No untyped escapes (`any`, bare `dict` passed around) without a written reason.
- Set up the standard linter + formatter for the stack during scaffold, and keep the tree clean.
- Explicit imports. No wildcard imports.
- Pin major versions. Every dependency earns its place; no "nice to have" installs. A few lines beats a new dependency.

## Naming

Follow the dominant convention of the language and framework, consistently:

| Element | Typical convention |
|---------|--------------------|
| Components / classes / types | PascalCase |
| Functions / variables | camelCase (JS/TS), snake_case (Python) |
| Constants | UPPER_SNAKE |
| Files | match the ecosystem (PascalCase components, kebab-case routes, snake_case Python modules) |
| DB columns | snake_case |
| Env variables | UPPER_SNAKE |

The point is consistency within the project, not a universal scheme. Pick the ecosystem norm and hold it.

## Structure

- Group by feature/domain for anything past trivial (`features/auth/`, `modules/payments/`), not by file-type buckets.
- One primary unit per file; file name matches the unit.
- Co-locate tests with source unless the ecosystem says otherwise.
- Keep the layering the architecture specifies (e.g. routes -> handlers -> services, or the framework's equivalent). No business logic in the transport layer.

## Error handling

- Never swallow errors silently. Log with context or propagate.
- Consistent error response shape across the API (e.g. `{ error, code, details? }`), documented in the architecture's contract.
- Async boundaries are wrapped (try/catch or the framework's error middleware). No unhandled rejections.
- On the client: every data call handles loading, error, and success. Error boundaries at least at the page level.

## Security baseline (not optional)

- **No secrets in source.** Environment variables only; ship `.env.example` with every required key and no real values.
- **Validate all input** at the boundary (a schema validator such as zod/pydantic), both client and server. Never trust client-supplied IDs, amounts, or roles.
- **Parameterized queries only.** Let the ORM/driver bind values; never string-concatenate SQL.
- **Authz on every mutation.** Re-check ownership/role server-side. If the blueprint uses row-level security, enforce it at the database and re-check on server mutations.
- **Payments/webhooks:** verify webhook signatures, use idempotency keys on money mutations, compute amounts server-side, handle refunds/disputes/failed states.
- **Escaping / XSS:** rely on the framework's default escaping; avoid raw HTML injection APIs.
- **Transport & tokens:** HTTPS assumed; keep sensitive tokens in httpOnly cookies or the Authorization header, not in localStorage.
- **CORS** explicitly configured; never `*` in production.
- Security-critical libraries (hashing, auth, headers) pinned to current stable.

## Comments

- Do not restate what the code says.
- Do comment: business-rule rationale, non-obvious algorithms, and workarounds (with the reason).
- Public/consumed APIs get a doc comment; internal glue does not need one.

## The polish bar (Phase 6)

Before a build is "done", ponytail/impeccable pass should leave: no dead code, no speculative abstraction, no duplicated logic that a shared helper would remove, no leftover scaffold/placeholder content, and names that read clearly. Behavior unchanged; the diff only removes and tightens. Re-run typecheck + tests after.
