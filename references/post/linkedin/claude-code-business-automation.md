Everyone's excited that Claude Code can now click through any app like a human.

That's the exact part I trust least in client work.

The flashy demo and the thing that actually survives production are two different features.

Computer Use lets an agent navigate a screen visually. Genuinely useful for legacy portals with no API.

But screen-clicking breaks the moment a button moves. I've watched "autonomous" flows fall apart on a single layout change, mid-run, with no clean way to recover.

The durable wins come from MCP. Structured read and write across your real tools. The inbox, the CRM, the sheet, the accounting system.

That's the difference between scraping a screen and calling an API. One is a magic trick. The other is infrastructure.

So when I scope an automation now, I ask one thing first: does this tool expose a real connection, or am I asking an agent to pretend to be a mouse?

If it's the mouse, I price in the breakage.

The teams getting value out of this aren't chasing the most autonomous demo. They picked one boring, repetitive task, proved it against the edge cases, then expanded.

"Start small" isn't safe advice. It's the only approach that compounds.

What's the one repetitive task you'd hand to an agent first?
